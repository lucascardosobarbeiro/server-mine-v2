GitOps-Powered Minecraft Server on GCP

This project provides a robust, scalable, and secure Infrastructure as Code (IaC) solution for deploying a production-ready PaperMC Minecraft server with a Velocity proxy on the Google Cloud Platform (GCP). It automates the entire lifecycle through a GitOps-centric CI/CD pipeline, from provisioning and security scanning to application deployment.

The core principle of this architecture is to allow for any modification‚Äîfrom infrastructure changes to plugin updates‚Äîto be managed, validated, and deployed automatically through Git, eliminating manual intervention.

üèõÔ∏è Architecture Details

This architecture is designed for automation and security, divided into three main layers: Infrastructure (GCP & Terraform), Application (Docker), and CI/CD (GitHub Actions).

1. Infrastructure Layer (Provisioning with Terraform on GCP)

Terraform declaratively defines and provisions all necessary GCP resources, ensuring the environment is reproducible and version-controlled.

  • Networking (VPC): A dedicated Virtual Private Cloud isolates the server environment. Firewall rules are configured to only allow necessary traffic.
  • Compute (Compute Engine): A single VM instance is provisioned to run the Docker containers.
  • Storage (Google Cloud Storage): Two GCS buckets are created: one for the remote Terraform state (`tfstate`) and another for world backups.
  • Security & Identity (IAM): A dedicated Service Account is created for the CI/CD pipeline. Workload Identity Federation allows the pipeline to securely authenticate with GCP without long-lived keys. SSH access is secured via IAP (Identity-Aware Proxy).

2. Application Layer (Containerization with Docker)

The Minecraft server is managed by Docker, ensuring a portable and consistent runtime.

  • docker-compose.yml: This file orchestrates the `papermc` and `velocity` services. The CI/CD pipeline syncs this file to the VM.
  • Volume Mounts: The `docker-compose.yml` file uses volume mounts to map configurations and plugins downloaded by the CI/CD pipeline directly into the running containers.

3. CI/CD & GitOps Layer (GitHub Actions)

This layer connects the code repository to the cloud infrastructure, enabling full automation based on Git operations. The .github/workflows/deploy.yml pipeline follows a security-first approach:

  1. Secure Authentication: The pipeline authenticates to GCP using its dedicated Service Account via Workload Identity Federation.
  2. Code Validation & Security Scan: Before any infrastructure is touched, the pipeline runs a series of critical checks.
  3. Infrastructure Deployment (terraform plan & apply): If validation passes, `terraform plan` is executed to create an execution plan, and `terraform apply` runs automatically to update the infrastructure on GCP if changes are needed.
  4. Application Deployment: The pipeline generates server configuration files, downloads plugins, and copies all necessary files to the VM, restarting the Docker containers to apply the changes.

‚öôÔ∏è Usage (CI/CD Driven)

This project is designed to be managed primarily through Git.

  1. Clone the repository.
  2. Configure GitHub Secrets: In your repository settings (Settings > Secrets and variables > Actions), create all the secrets required by the deploy.yml workflow.
  3. Make a change: Edit a .tf file, update a plugin in manifest.json, or change a server configuration template.
  4. Commit and Push: 
    git add .
git commit -m "Your descriptive change message"
git push origin main
  5. Monitor the Pipeline: Go to the "Actions" tab in your GitHub repository to watch the pipeline validate, test, and deploy your changes automatically.
